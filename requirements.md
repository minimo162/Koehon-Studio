# requirements.md

# AI原稿対応オーディオブック作成アプリ 要件定義

## 1. 概要

本アプリは、外部AIモデルで作成した Markdown / txt 形式の音声用原稿を読み込み、ローカル環境で音声化し、章単位または作品単位の音声ファイルとして出力する Windows 向けデスクトップアプリである。

ユーザーは、PDF、Word、PowerPoint、Webページ、議事録、メール、チャットログ、マニュアル、メモなどの任意の元資料を、M365 Copilot、ChatGPT、Claude などの外部AIに渡して、音声で聞きやすい Markdown 原稿に変換する。その後、本アプリに原稿を読み込ませ、MOSS-TTS-Nano 系のTTSエンジンを使って音声化する。

本アプリ自体は、初期段階では元資料の解析を直接行わない。元資料から原稿を作る工程は外部AIに任せ、本アプリは「原稿の読み込み」「章分割」「読み上げ用チャンク分割」「音声生成」「プレビュー」「書き出し」に集中する。

## 2. 前提

- 対象OSは Windows 10 / Windows 11 とする。
- UIは Tauri + Svelte + TypeScript で実装する。
- TTS推論は、ONNX Runtime を利用するネイティブ sidecar に分離する。
- sidecar はユーザー環境に Python、pip、venv、Node.js、Rust などの追加ランタイム導入を要求しない standalone 配布物とする。
- 初期TTSエンジンは MOSS-TTS-Nano 系の ONNX 版を想定する。
- アプリは原則としてローカル動作とし、原稿本文や生成音声を外部サーバーへ送信しない。
- 外部AIによる原稿生成は、アプリ外の作業として扱う。
- アプリには、外部AIへ渡すための汎用プロンプトテンプレートを表示・コピーする機能を持たせる。

## 3. 目的

### 3.1 主目的

任意の元資料から作られた音声用原稿を、ユーザーが簡単にオーディオブック形式の音声へ変換できるようにする。

### 3.2 副目的

- PDFに限定しない汎用的な原稿ワークフローを提供する。
- TTS前に原稿を確認・編集できるようにする。
- 長文でも章単位・段落単位で安定して音声化できるようにする。
- 失敗したチャンクだけを再生成できるようにする。
- 将来的に EPUB、DOCX、PDF、HTML などの直接取り込みに拡張できる構成にする。

## 4. 非目的

初期段階では以下を対象外とする。

- 元資料そのものの高度な解析。
- PDF OCR。
- Word、PowerPoint、Excel の直接解析。
- 外部AI APIを使った原稿自動生成。
- 商用クラウドTTSとの連携。
- 複数話者による高度な会話劇生成。
- 音声波形編集ソフト並みの詳細編集。
- DRM付きコンテンツの変換。

## 5. 想定ユーザー

### 5.1 個人学習者

研修資料、論文、記事、メモなどを音声化し、移動中や作業中に聞きたいユーザー。

### 5.2 社内研修担当者

社内資料や研修スライドを、聞きやすい音声教材として作成したいユーザー。

### 5.3 コンテンツ制作者

自作原稿や資料をもとに、簡易的なオーディオブックやナレーション音声を作りたいユーザー。

### 5.4 業務効率化ユーザー

会議メモ、議事録、メール要約、プロジェクト資料などを音声で確認したいユーザー。

## 6. 基本ワークフロー

```text
任意の元資料
  ↓
外部AIで「オーディオブック用ナレーション原稿」を作成
  ↓
Markdown / txt 原稿として保存
  ↓
本アプリで原稿を読み込み
  ↓
原稿プレビュー・章分割・チャンク分割
  ↓
TTSで音声生成
  ↓
章ごとの音声確認
  ↓
WAV / MP3 / M4B などで書き出し
```

## 7. 対応原稿形式

### 7.1 初期対応

- `.md`
- `.txt`

### 7.2 将来対応候補

- `.json`
- `.docx`
- `.pdf`
- `.html`
- `.epub`

## 8. 標準原稿フォーマット

アプリは以下の Markdown 形式を標準原稿として扱う。

```markdown
---
title: 生成されたタイトル
source_type: PowerPoint
source: 2026年度 研修資料
audience: 新入社員
language: ja-JP
style: 研修講師風
version: 1
---

# はじめに

このオーディオブックでは、元資料の内容をもとに重要なポイントを解説します。

[pause:medium]

# 第1章 背景

本文……

[pause:medium]

# 第2章 重要な考え方

本文……

[pause:medium]

# まとめ

本文……

[pause:long]

# 原稿作成メモ

## 省略した内容

本文……

## 確認が必要な点

本文……
```

## 9. 特殊タグ

初期実装では以下のタグを解釈する。

| タグ | 意味 | 既定値 |
|---|---|---:|
| `[pause:short]` | 短い無音 | 0.5秒 |
| `[pause:medium]` | 標準の無音 | 1.0秒 |
| `[pause:long]` | 長い無音 | 2.0秒 |
| `[chapter:end]` | 章の明示的終了 | なし |
| `[voice:名前]` | 話者指定 | 初期実装では任意 |
| `[speed:値]` | 読み上げ速度指定 | 初期実装では任意 |

初期実装では `[pause:*]` を必須対応とし、`[voice:*]` と `[speed:*]` は構文保持のみでもよい。

## 10. 機能要件

### R1. 原稿読み込み

ユーザーは、Markdown またはテキストファイルをアプリに読み込める。

受け入れ条件:

- WHEN ユーザーが `.md` または `.txt` ファイルを選択する THEN アプリはファイル内容を読み込む。
- WHEN ファイルに YAML front matter が含まれる THEN アプリはタイトル、元資料種別、対象読者、言語、スタイルなどのメタ情報を抽出する。
- WHEN ファイルに front matter がない THEN アプリは本文から仮タイトルを生成する、または「無題」として扱う。
- WHEN ファイルの文字コードが UTF-8 の場合 THEN アプリは文字化けなく読み込む。
- WHEN 読み込みに失敗する THEN アプリは失敗理由を表示する。

### R2. 原稿プレビュー

ユーザーは、読み込んだ原稿をアプリ上で確認できる。

受け入れ条件:

- WHEN 原稿が読み込まれる THEN アプリは本文プレビューを表示する。
- WHEN Markdown見出しがある THEN アプリは章一覧を表示する。
- WHEN `[pause:*]` タグがある THEN アプリはプレビュー上で無音タグとして分かりやすく表示する。
- WHEN ユーザーが章を選択する THEN アプリは該当章の本文を表示する。

### R3. 原稿編集

ユーザーは、読み込んだ原稿を音声化前に編集できる。

受け入れ条件:

- WHEN ユーザーが本文を編集する THEN アプリは編集内容を保持する。
- WHEN ユーザーが保存を実行する THEN アプリは編集後の原稿を保存する。
- WHEN 未保存の変更がある状態で閉じようとする THEN アプリは確認ダイアログを表示する。

### R4. 章分割

アプリは、Markdown見出しをもとに原稿を章単位に分割する。

受け入れ条件:

- WHEN `#` 見出しが存在する THEN アプリは `#` 見出しを章として扱う。
- WHEN `##` 以下の見出しが存在する THEN アプリは章内セクションとして扱う。
- WHEN 見出しが存在しない THEN アプリは全文を単一章として扱う。
- WHEN `# 原稿作成メモ` が存在する THEN 初期設定では読み上げ対象から除外する。

### R5. チャンク分割

アプリは、TTSエンジンに渡しやすい長さへ本文を分割する。

受け入れ条件:

- WHEN 章本文が長い THEN アプリは句点、改行、段落、文字数を考慮してチャンクへ分割する。
- WHEN チャンク内に `[pause:*]` がある THEN アプリは無音チャンクとして扱う。
- WHEN 1チャンクが最大文字数を超える THEN アプリは文境界を優先して分割する。
- WHEN 文境界が見つからない THEN アプリは最大文字数で強制分割する。

### R6. TTS sidecar 起動

アプリは、同梱されたローカルTTS sidecarを起動・停止できる。

受け入れ条件:

- WHEN アプリ起動時にTTS sidecar が未起動 THEN アプリは同梱されたネイティブ sidecar を起動する。
- WHEN sidecar が起動した THEN アプリは `/health` で接続確認する。
- WHEN ユーザー環境に Python が存在しない THEN アプリは通常どおり音声生成できる。
- WHEN sidecar 起動に失敗した THEN アプリはエラー内容と対処案を表示する。
- WHEN アプリ終了時 THEN アプリは sidecar を停止する。

### R7. 音声生成

ユーザーは、原稿全体または章単位で音声を生成できる。

受け入れ条件:

- WHEN ユーザーが「全体を生成」を実行する THEN アプリは全章を順番に生成キューへ追加する。
- WHEN ユーザーが「章を生成」を実行する THEN アプリは選択章のみを生成する。
- WHEN TTSチャンク生成が完了する THEN アプリは一時音声ファイルを保存する。
- WHEN 生成に失敗したチャンクがある THEN アプリは該当チャンクを失敗状態として表示する。
- WHEN 失敗チャンクがある THEN ユーザーはそのチャンクだけを再生成できる。

### R8. 進捗表示

アプリは音声生成の進捗を表示する。

受け入れ条件:

- WHEN 音声生成中 THEN アプリは章単位とチャンク単位の進捗を表示する。
- WHEN 現在生成中のチャンクがある THEN アプリは対象章、チャンク番号、本文冒頭を表示する。
- WHEN 生成が完了した THEN アプリは完了通知を表示する。
- WHEN ユーザーが停止を押す THEN アプリは現在のキューを停止する。

### R9. 音声プレビュー

ユーザーは、生成した音声をアプリ内で再生できる。

受け入れ条件:

- WHEN 章の音声が生成済み THEN アプリは章単位の再生ボタンを表示する。
- WHEN チャンク音声が生成済み THEN アプリはチャンク単位の再生を可能にする。
- WHEN 再生中に別の音声を選択する THEN アプリは現在の再生を停止し、新しい音声を再生する。

### R10. 音声結合

アプリは、生成済みのチャンク音声を章単位または全体単位で結合する。

受け入れ条件:

- WHEN 章内の全チャンク生成が完了している THEN アプリは章音声を結合できる。
- WHEN `[pause:*]` が含まれる THEN アプリは指定秒数の無音を挿入する。
- WHEN 全章の生成が完了している THEN アプリは全体音声を結合できる。

### R11. 書き出し

ユーザーは生成音声を書き出せる。

受け入れ条件:

- WHEN ユーザーが WAV 書き出しを選択する THEN アプリは WAV ファイルを保存する。
- WHEN MP3 書き出しが利用可能 THEN アプリは MP3 ファイルを保存する。
- WHEN M4B 書き出しが利用可能 THEN アプリは M4B ファイルを保存する。
- WHEN 章別書き出しを選ぶ THEN アプリは章ごとのファイルを保存する。
- WHEN 全体書き出しを選ぶ THEN アプリは1つのファイルとして保存する。

### R12. プロジェクト保存

ユーザーは作業状態をプロジェクトとして保存・再開できる。

受け入れ条件:

- WHEN ユーザーがプロジェクト保存を実行する THEN アプリは原稿、メタ情報、生成状況、音声ファイル参照を保存する。
- WHEN ユーザーがプロジェクトを開く THEN アプリは前回の状態を復元する。
- WHEN 音声ファイルが見つからない THEN アプリは欠落ファイルを表示し、再生成を促す。

### R13. AI原稿生成プロンプト表示

ユーザーは、外部AIに渡す原稿生成プロンプトをアプリからコピーできる。

受け入れ条件:

- WHEN ユーザーが「プロンプト」画面を開く THEN アプリは汎用プロンプトを表示する。
- WHEN ユーザーが元資料種別、対象読者、文体、長さを選ぶ THEN アプリは条件を反映したプロンプトを生成する。
- WHEN ユーザーがコピーを押す THEN アプリはプロンプトをクリップボードへコピーする。
- WHEN 元資料種別が PowerPoint、Excel、議事録、Webページなどの場合 THEN アプリは種別別の追加指示を反映できる。

### R14. 設定

ユーザーはTTSと出力に関する基本設定を変更できる。

受け入れ条件:

- WHEN ユーザーが設定画面を開く THEN アプリはTTSエンジン、話者、CPUスレッド数、チャンク長、出力形式を表示する。
- WHEN ユーザーが設定を変更する THEN アプリは変更を保存する。
- WHEN 設定値が不正 THEN アプリは保存前に警告する。

### R15. ログ

アプリはトラブルシュート用ログを保持する。

受け入れ条件:

- WHEN 生成処理を実行する THEN アプリは処理ログを保存する。
- WHEN エラーが発生する THEN アプリはエラー種別、対象チャンク、時刻を記録する。
- WHEN ユーザーがログ表示を開く THEN アプリは直近ログを確認できる。

## 11. 非機能要件

### N1. ローカル性

- 原稿本文、生成音声、プロジェクトデータはローカルに保存する。
- アプリはTTS生成のために外部サーバーへ本文を送信しない。
- 外部AIを利用する場合はユーザーがアプリ外で実施する。
- TTS推論は同梱されたネイティブ sidecar 内で実行する。

### N2. パフォーマンス

- 長文原稿でもUIが固まらないよう、生成処理は非同期で行う。
- 生成キューの停止・再開が可能であること。
- チャンク単位で途中結果を保存し、失敗時に全体をやり直さなくてよいこと。

### N3. 安定性

- TTS sidecar がクラッシュしても、アプリ本体は終了しない。
- クラッシュ後は再起動を試行できる。
- 生成済みチャンクは保持する。
- sidecar はONNX Runtime初期化失敗、モデル不足、DLL不足を区別して報告する。

### N4. セキュリティ

- sidecar は原則として `127.0.0.1` のみに bind する。
- 必要に応じてローカルAPIトークンを使用する。
- 任意コマンド実行につながる設定値をユーザー入力から直接渡さない。
- 原稿内タグはホワイトリスト方式で解釈する。
- ユーザー指定パスを sidecar 起動コマンドとして直接実行しない。

### N5. 保守性

- UI、Tauri core、TTS sidecar を分離する。
- TTSエンジン差し替えを想定した抽象インターフェースを設ける。
- 原稿パーサーはユニットテスト可能な純粋関数として実装する。
- sidecar 内では、ONNX Runtime 推論層、モデル管理層、HTTP/IPC API層、WAV出力層を分離する。

### N6. アクセシビリティ

- キーボード操作で主要機能を使えるようにする。
- 進捗やエラーは視覚的に分かりやすく表示する。
- 長文編集画面は読みやすいフォントサイズにする。

## 13. 原稿生成プロンプト要件

アプリが提供するプロンプトは、以下の方針を満たす。

- PDFに限定せず「元資料」と表現する。
- 外部AIに対して、元資料だけを根拠にするよう指示する。
- 音声だけで理解できる原稿に再構成するよう指示する。
- 表、図、スライド、箇条書き、URL、脚注を自然なナレーションに変換するよう指示する。
- Markdown形式で出力させる。
- `[pause:short]`、`[pause:medium]`、`[pause:long]` を使わせる。
- 「原稿作成メモ」に省略内容と確認事項を出させる。

## 14. 初期実装範囲

初期実装では以下を実装する。

- Markdown / txt 原稿読み込み。
- 原稿プレビュー。
- 見出しベースの章分割。
- `[pause:short]`、`[pause:medium]`、`[pause:long]` 解釈。
- チャンク分割。
- ONNX Runtime を利用するネイティブ TTS sidecar 起動。
- ユーザー環境にPythonを要求しない配布。
- 章単位・全体単位の音声生成。
- チャンク単位の失敗表示と再生成。
- WAV 書き出し。
- AI原稿生成プロンプトの表示・コピー。
- 設定保存。

## 16. リスク

| リスク | 内容 | 対策 |
|---|---|---|
| TTS品質 | 日本語長文の読み間違いが起きる | 原稿正規化、読み替え辞書、チャンク再生成 |
| 生成速度 | CPU推論が遅い可能性 | チャンク単位進捗、GPU対応検討、設定可能なCPUスレッド |
| 配布サイズ | モデル同梱でサイズが大きい | 初回ダウンロード方式、モデル管理画面 |
| AI原稿の誤り | 外部AIが誤要約する | 原稿プレビュー、原稿作成メモ、確認事項表示 |
| 権利問題 | 元資料や声の利用権利 | 注意表示、利用者責任の明示 |
| sidecar障害 | ネイティブ sidecar が落ちる、ONNX Runtime 初期化に失敗する | health check、再起動、ログ保存、モデル/DLL診断 |

## 17. 完了条件

初期実装の完了条件は以下とする。

- ユーザーが Markdown 原稿を読み込める。
- 章一覧が表示される。
- `[pause:*]` を含む原稿を音声化できる。
- 章単位のWAVを生成できる。
- 全体WAVを書き出せる。
- 失敗チャンクを表示し、再生成できる。
- 外部AI用プロンプトをコピーできる。
- アプリを終了して再起動してもプロジェクトを再開できる。
- Pythonが入っていないWindows環境でもTTS sidecarを起動して音声生成できる。
